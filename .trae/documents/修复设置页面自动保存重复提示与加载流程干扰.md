**问题分析**
- 切换页面返回后仍弹“设置已自动保存”，原因是：在 `loadSettings()` 里给表单模型赋值会触发我们对这些字段的 `watch`，从而误判为“用户改动”，再次调用 `saveSettings()` 并弹通知。
- 另一个隐患是：`saveSettings()` 不区分“用户触发”与“初始化/程序设置”，会对主进程做副作用操作（托盘、开机自启）并提示，从而造成不必要的噪声。

**修复方案**
1. 引入“加载中”保护标志：在设置加载期间禁用自动保存
   - 在 `<script setup>` 中添加 `const isLoadingConfig = ref(false)`。
   - 在 `loadSettings()` 开始时设为 `true`，完成赋值后设为 `false`。
   - 所有与设置项绑定的 `watch(...)` 回调开头增加：`if (isLoadingConfig.value) return;`。
   - 作用：避免初始化赋值导致的误触发，彻底消除切页返回后的重复“自动保存”提示。

2. 为自动保存增加抑制提示的开关与轻微节流
   - 将 `saveSettings()` 改为接受参数：`saveSettings({ silent?: boolean } = {})`。
   - 当 `silent === true` 时不调用 `MessagePlugin.success`。
   - 在各个 `watch` 中调用 `saveSettings({ silent: false })`（用户实际操作），而在程序性同步（例如从 `getUserDataDir()` 回写 `config.dir`）时使用 `saveSettings({ silent: true })`。
   - 增加 300ms 节流（例如用局部 `setTimeout`/清除前一个触发）以防快速连改产生多次保存与多次提示。

3. 区分“用户交互”与“程序写入”的保存触发点
   - 对需要即时副作用的开关（`minimizeToTray`、`autoStart`）保留 `watch`，但在 `isLoadingConfig` 为 `true` 时不触发。
   - 对数值/文本类设置（如端口、路径）在输入组件的 `@change` / `@blur` 事件触发保存更干净：
     - `t-input-number` 使用 `@change="onPortChange"`，内部调用 `saveSettings({ silent: false })`。
     - 路径选择按钮点击后更新 `configDir` 并调用 `saveSettings({ silent: false })`。
   - 作用：将“提示”严格限定在用户交互产生的保存。

4. 轻量校验与一致性优化
   - `serverPort` 在保存前做范围校验并回退到上次有效值，避免错误值触发保存。
   - `configDir` 为空时不保存，保持当前值。

5. 验证与回归检查
   - 进入设置页：仅加载，不弹提示。
   - 修改任一设置：立即自动保存并提示一次。
   - 切换到其他页面再返回：不弹提示；各设置值维持已保存状态。
